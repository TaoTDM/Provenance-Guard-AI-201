# Provenance Guard вҖ” planning.md

A backend service any creative-sharing platform can plug in to estimate whether submitted text was
written by a human or generated by AI, communicate that estimate **honestly** (with calibrated
uncertainty), label it for readers, and let creators appeal. Built before implementation per Milestone 2.

> Design principle that drives every decision below: **a false positive (calling a human's work AI)
> is worse than a false negative.** Thresholds, labels, and the appeal path are all biased to protect creators.

---

## 1. Detection Signals

The pipeline runs **three independent signals** (3 satisfies the *Ensemble detection* stretch; the project
only requires 2). Each returns a probability-of-AI `p in [0,1]`, so they're combinable on a common scale.

| # | Signal | Type | What it measures | Output | Blind spot |
|---|--------|------|------------------|--------|------------|
| 1 | **Groq LLM classifier** (`llama-3.3-70b-versatile`) | Semantic | Holistic "does this read as human or AI?" вҖ” coherence, register, idea flow | `p_llm вҲҲ [0,1]` + short rationale (JSON) | Confidently wrong on polished human prose; non-deterministic; can be prompt-gamed |
| 2 | **Stylometric heuristics** | Structural / statistical | Sentence-length *variance* (burstiness), type-token ratio (vocab diversity), punctuation density, mean sentence length | `p_style вҲҲ [0,1]` (weighted blend of normalized metrics) | Penalizes formal/non-native human writing; unstable on short text |
| 3 | **Lexical-register heuristic** | Surface / formulaic | Density of AI-clichГ© transitions ("furthermore", "it is important to note", "paradigm shift", "in conclusion") + contraction rate + function-word ratio | `p_lex вҲҲ [0,1]` | Misses AI text that's been deliberately de-formalized; can flag academic humans |

**Why these three:** signal 1 is semantic, signal 2 is statistical-structural, signal 3 is surface-lexical вҖ”
three *genuinely different views* of the same text, so agreement is informative and disagreement is a real signal.

**Combination (ensemble weighting + disagreement voting):**
```
p_ai = 0.50 * p_llm + 0.30 * p_style + 0.20 * p_lex      # LLM trusted most
disagreement = stdev([p_llm, p_style, p_lex])            # 0 = perfect agreement
```
Weights live in a `WEIGHTS` constant for easy recalibration. If the Groq call fails, the system **degrades
gracefully**: re-normalize weights over signals 2 & 3, set a `degraded: true` flag, and cap confidence (logged).

## 2. Uncertainty Representation

**What the numbers mean (decided before implementing):**
- `p_ai` вҖ” combined probability the text is AI-generated.
- `confidence` вҖ” how *sure* we are of the verdict, **independent of direction**:
  `confidence = 2 * |p_ai - 0.5| * (1 - disagreement)`
  вҶ’ `0.0` = a coin-flip / signals fighting each other; `1.0` = unanimous and extreme.
  The disagreement term is the key honesty mechanism: when signals contradict, confidence drops toward 0.

**Asymmetric thresholds (encodes the false-positive priority вҖ” wider gate before we accuse a human):**

| Condition | Verdict | Rationale |
|-----------|---------|-----------|
| `p_ai вүҘ 0.70` and `confidence вүҘ 0.50` | `likely_ai` | High bar: we only call AI on strong, agreeing evidence |
| `p_ai вүӨ 0.35` | `likely_human` | Easier to clear someone than to accuse them |
| everything else (incl. all low-confidence cases) | `uncertain` | When unsure, we say so |

**Short-text guard:** if `word_count < 40`, force `uncertain` (statistical signals are unreliable) and note `low_evidence`.

So `0.6` to this system means: "leaning AI but inside the uncertain band вҖ” we are *not* making an AI claim."
A `0.95` lands firmly in `likely_ai`; a `0.51` lands in `uncertain`. Different bands вҶ’ different labels (not a flip at 0.5).

**How we'll validate it's meaningful (Milestone 4 test plan):** run the 4 provided calibration texts
(clear-AI, clear-human, formal-human, lightly-edited-AI) + a repetitive poem + a 1-line text, and assert the
*ordering* of scores and that each of the three label variants is reachable. Committed as `tests/test_calibration.py`.

## 3. Transparency Label вҖ” Three Variants (verbatim)

`{confidence}` is rendered as a whole-number percent. Every label states it's an automated estimate; the AI
label always invites appeal (false-positive safeguard).

**High-confidence AI (`likely_ai`):**
> рҹӨ– **Likely AI-generated** вҖ” Our analysis suggests this content was probably created with help from AI
> tools (estimated confidence: {confidence}%). This is an automated estimate, not a verdict of fact. If
> you're the creator and disagree, you can appeal this assessment.

**High-confidence human (`likely_human`):**
> вңҚпёҸ **Likely human-written** вҖ” Our analysis found no strong signs of AI generation in this content
> (estimated confidence: {confidence}%). This is an automated estimate and can still be wrong.

**Uncertain (`uncertain`):**
> вқ“ **Attribution uncertain** вҖ” We could not confidently determine whether this content was written by a
> human or generated by AI. We're telling you this honestly rather than guessing. **No AI-generation claim
> is being made about this work.**

## 4. Appeals Workflow

- **Who:** the content's creator (matched on `creator_id` from the original submission).
- **They provide:** `content_id` + `creator_reasoning` (free text; e.g. "I'm a non-native speaker, my style reads formal").
- **System does (`POST /appeal`):**
  1. Look up `content_id`; reject if unknown.
  2. Update content `status` вҶ’ `under_review` (SQLite).
  3. Write an `appeal` audit entry that references the **original decision snapshot** (verdict, confidence, all 3 signal scores).
  4. Return confirmation: `appeal_id`, new status, and the original decision so the creator sees what they're contesting.
- **No automated re-classification** (per spec) вҖ” an appeal flags for a human.
- **Reviewer queue (`GET /appeals`):** lists every `under_review` item with original text, verdict, confidence,
  the three signal scores, the creator's reasoning, and timestamps вҖ” everything a human needs to adjudicate.

---

## 5. Anticipated Edge Cases

1. **Non-native / formal human writing** вҖ” low burstiness + formal register + heavy transitions makes signals
   2 & 3 over-flag as AI. This is the classic false positive. Mitigation: LLM weighted highest, wide uncertain
   band, and the appeal path (the provided test reasoning is exactly this case).
2. **Repetitive simple poetry** (villanelle, nursery rhyme) вҖ” deliberately low type-token ratio + uniform line
   length looks "uniform like AI" to stylometrics. Mitigation: uncertain band + LLM semantic check.
3. **Very short text (< 40 words)** вҖ” variance metrics are statistically meaningless. Mitigation: short-text guard вҶ’ forced `uncertain`.
4. **Lightly-edited AI / human-AI collaboration** вҖ” genuinely in-between; *should* land `uncertain`, and that's the honest answer, not a failure.

`under_review` вҶ’ writes an appeal audit entry linked to the original decision вҶ’ returns confirmation with the

## Rate Limiting (decided here, documented in README)

`@limiter.limit("10 per minute; 100 per day")` on `/submit`, in-memory storage (`storage_uri="memory://"`),
keyed on remote address.
**Reasoning:** a real creator submits their own work occasionally вҖ” even an active writer rarely posts >100
pieces/day, and bursts above ~10/min indicate scripted abuse, not authorship. The per-minute cap stops flooding;
the daily cap stops slow-drip scraping. `/appeal` gets a stricter `5 per minute` (appeals are rare and human-paced).

## Stretch Features (all four вҖ” designed now per the "update planning.md before starting" rule)

**S1 В· Ensemble detection** вҖ” *already the core design*: 3 signals, documented weights (0.50/0.30/0.20) +
disagreement-based confidence damping (a soft vote: contradiction вҶ’ lower confidence вҶ’ `uncertain`).

**S2 В· Provenance certificate ("Verified Human")** вҖ” `POST /verify`: creator completes an extra step
(timed original-passage challenge, simplest viable form) вҶ’ system issues a `certificate_id` stored against
`creator_id` (`certificates` table: creator_id, cert_id, method, issued_at). Future submissions by that creator
append a **вң… Verified Human Creator** badge to the label; detection still runs (badge вү  exemption) but the badge
is surfaced to readers as provenance context.

**S3 В· Analytics dashboard** вҖ” `GET /analytics` (JSON) + a minimal HTML view: total submissions, verdict
distribution, **appeal rate**, average confidence, signal-agreement rate, and the extra metric of choice =
**AI-verdict appeal rate** (a proxy for false-positive rate вҖ” directly ties back to the design principle).

**S4 В· Multi-modal support** вҖ” add a `content_type` field (`text` | `image_metadata`). For `image_metadata`,
accept structured fields (generator/software tag, C2PA presence, prompt text) + a description string; run a
**metadata signal** (e.g. generator tag present / C2PA absent) plus the text signals on the description.
Pipeline branches on `content_type`; same scoring/label/appeal machinery downstream.

## Improvements Beyond Spec (documented as deliberate engineering)
- **Disagreement-aware confidence** вҖ” signals voting against each other can't yield high confidence.
- **Asymmetric thresholds** вҖ” structurally harder to accuse a human than to clear one (the hint, made concrete).
- **Short-text guard** вҖ” forces `uncertain` under 40 words instead of emitting garbage scores.
- **Graceful Groq degradation** вҖ” API outage re-normalizes to signals 2+3, flags `degraded`, caps confidence.
- **Config block** for weights/thresholds вҶ’ single place to recalibrate.
- **PII-safe logging** (hash + preview) and **content de-dup** via `text_hash`.
- **`tests/test_calibration.py`** вҖ” asserts score ordering across labeled samples so regressions are caught.
- **`GET /healthz`** вҖ” liveness + reports Groq reachability / degraded mode.
