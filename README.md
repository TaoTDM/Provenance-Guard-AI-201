# Provenance Guard

A backend service that any creative-sharing platform can plug in to estimate whether a piece of
submitted content was **written by a human or generated by AI** — and, crucially, to communicate that
estimate *honestly*. It scores confidence, surfaces a plain-language transparency label to readers, and
gives creators a path to appeal a verdict they disagree with.

> **Design principle that drives every decision here:** on a writing platform, a **false positive
> (labelling a human's work as AI) is worse than a false negative.** Thresholds, labels, and the appeal
> path are all biased toward protecting creators. Perfect AI detection is an unsolved problem; the
> engineering goal is to *acknowledge uncertainty* rather than force a binary verdict.

Built for AI201 Project 4. The pre-implementation design lives in [planning.md](planning.md).

## Quick start

```bash
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt

echo "GROQ_API_KEY=your_key_here" > .env   # your Groq key (free tier)

python app.py                       # serves on http://127.0.0.1:5000
```

Submit a piece of text:

```bash
curl -s -X POST http://127.0.0.1:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "The sun dipped below the horizon, painting the sky in amber and rose.", "creator_id": "test-user-1"}' \
  | python -m json.tool
```

Run the calibration tests (offline, no API spend):

```bash
python -m tests.test_calibration
```

## Architecture overview

**The path a submission takes from input to transparency label:**

```
Creator ──text + creator_id──> POST /submit
                                   │
                                   ▼
                        ┌──────── Detection Pipeline ────────┐
   Signal 1: Groq LLM ──┤ p_llm                              │
   Signal 2: Stylometry ┤ p_style  ──> weighted blend ──> p_ai + disagreement
   Signal 3: Lexical ────┤ p_lex                             │
                        └────────────────────────────────────┘
                                   │
                                   ▼
                    Confidence scoring (asymmetric bands,
                    disagreement override, short-text guard)
                                   │  verdict + confidence
                                   ▼
                    Label generator ── one of 3 variants ──┐
                                   │                        │
                                   ▼                        ▼
                       Audit log (SQLite)  ◀───  JSON response to caller
```

1. **`POST /submit`** receives raw `text` (or image metadata) and a `creator_id`.
2. The **detection pipeline** runs three independent signals, each returning a probability the content is
   AI-generated (`p ∈ [0,1]`).
3. **Confidence scoring** blends them into a single `p_ai`, measures how much the signals *disagree*, and
   derives a calibrated `confidence`, then applies asymmetric verdict thresholds → `likely_ai`,
   `likely_human`, or `uncertain`.
4. The **label generator** maps that verdict to one of three plain-language transparency labels.
5. Everything (signal scores, verdict, confidence, timestamp) is written to a structured **SQLite audit
   log**, and the verdict + label is returned to the caller as JSON.

An appeal follows a second path: **`POST /appeal`** → status flips to `under_review` → the appeal is logged
alongside the original decision → confirmation returned. The full diagram and narrative are in
[planning.md → Architecture](planning.md#architecture).

## Detection signals

The pipeline uses **three independent signals** (the project requires at least two; three satisfies the
*Ensemble* stretch). They are "distinct" because each captures a genuinely different property of the text —
one semantic, one structural, one lexical — so their *agreement* is informative and their *disagreement* is
itself a usable signal.

| # | Signal | What it measures | Why it differs human vs. AI | What it misses (blind spot) |
|---|--------|------------------|------------------------------|------------------------------|
| 1 | **Groq LLM** (`llama-3.3-70b-versatile`) | Holistic semantic read: coherence, idea flow, originality, formulaic phrasing | AI prose is fluent but often generic, hedged, and templated; a model can recognise its own register | Non-deterministic; can be confidently wrong; will over-flag *formal* human writing unless explicitly told not to |
| 2 | **Stylometric heuristics** | Sentence-length **burstiness** (coefficient of variation) and mean sentence length | Human writing varies its rhythm (long, short, fragment); AI output is more uniform | Short texts (few sentences) make variance meaningless; *formal human* writing is also uniform → false positives |
| 3 | **Lexical-register heuristics** | Density of AI-cliché transitions ("furthermore", "it is important to note"), contraction rate, informal markers | AI leans on connective filler and avoids contractions/slang; humans use voice, contractions, emphasis | Misses AI text that's been deliberately de-formalised; can flag academic humans who write formally |

Each signal returns a probability-of-AI in `[0,1]`. **Type–token ratio (vocabulary diversity) was tested
and deliberately dropped** from the stylometric score — on the short submissions this system targets, it
sat around 0.85–0.88 for *both* human and AI text, so it carried no discriminating signal. It is still
computed and reported in the audit metrics for transparency.

## Confidence scoring

**How the signals are combined** (constants in [config.py](config.py)):

```
p_ai          = 0.50 · p_llm + 0.30 · p_style + 0.20 · p_lex      # LLM trusted most
disagreement  = stdev(p_llm, p_style, p_lex)                       # 0 = unanimous
confidence    = 2 · |p_ai − 0.5| · (1 − disagreement)             # 0 = coin-flip, 1 = certain
```

The **disagreement term is the honesty mechanism**: when the three signals contradict each other, confidence
is pulled toward zero even if the average looks decisive. Confidence is also the reason `0.51` and `0.95`
produce different labels — it is a continuous value, not a binary flip at `0.5`.

**Verdict thresholds (asymmetric on purpose — harder to accuse than to clear):**

| Condition | Verdict |
|-----------|---------|
| `disagreement > 0.33` (signals at war) | `uncertain` |
| `word_count < 40` (short-text guard) | `uncertain` |
| `p_ai ≥ 0.65` | `likely_ai` |
| `p_ai ≤ 0.38` | `likely_human` |
| otherwise | `uncertain` |

The AI gate (`0.65`, i.e. 0.15 above the midpoint) is stricter than the human gate (`0.38`, 0.12 below),
so borderline content errs toward *not* accusing a creator.

**How I validated the scores are meaningful.** I ran the project's labelled calibration corpus through the
pipeline and confirmed (a) the three verdict buckets are all reachable, (b) clearly-AI and clearly-human
text produce *noticeably* different scores, and (c) the documented thresholds actually separate the
categories. This is locked in as a regression test in [tests/test_calibration.py](tests/test_calibration.py)
(14 deterministic checks, no API calls).

**Two example submissions with noticeably different confidence (actual scores):**

| Case | Input (abridged) | Verdict | `p_ai` | **confidence** | `p_llm` / `p_style` / `p_lex` |
|------|------------------|---------|--------|----------------|-------------------------------|
| **High-confidence** | "ok so i finally tried that new ramen place downtown and honestly? underwhelming…" | `likely_human` | 0.12 | **0.67** | 0.10 / 0.04 / 0.30 |
| **Lower-confidence** | "The relationship between monetary policy and asset price inflation has been extensively studied…" | `uncertain` | 0.55 | **0.09** | 0.40 / 0.81 / 0.55 |

The casual human review is unanimous across signals → high confidence. The formal academic paragraph
splits the signals (stylometrics says "uniform = AI", the LLM says "human"), so the system honestly reports
near-zero confidence and refuses to make a claim — exactly the behaviour the design intends.

## Transparency label

Three variants, one per verdict, in plain language (no jargon — a non-technical reader understands them).
The label **visibly differs** between confidence levels: the AI and human variants name a likelihood and the
AI variant invites appeal; the uncertain variant makes *no claim at all*.

The percentage shown is the **directional likelihood** (`round(p_ai·100)` for AI, `round((1−p_ai)·100)` for
human) rather than the internal `confidence` value — see [Spec reflection](#spec-reflection) for why.

| Variant | Exact text displayed |
|---------|----------------------|
| **High-confidence AI** (`likely_ai`) | 🤖 Likely AI-generated — Our automated analysis estimates a {N}% likelihood that this content was created with help from AI tools. This is an automated estimate, not a verdict of fact, and it can be wrong. If you are the creator and disagree, you can appeal this assessment. |
| **High-confidence human** (`likely_human`) | ✍️ Likely human-written — Our automated analysis found no strong signs of AI generation in this content (about a {N}% likelihood it was written by a human). This is an automated estimate and can still be wrong. |
| **Uncertain** (`uncertain`) | ❓ Attribution uncertain — We could not confidently determine whether this content was written by a human or generated by AI, so we are not making a claim either way. We are telling you this honestly rather than guessing. No AI-generation claim is being made about this work. |

**Rendered examples (actual output):**

> 🤖 Likely AI-generated — Our automated analysis estimates a **73%** likelihood that this content was
> created with help from AI tools. This is an automated estimate, not a verdict of fact, and it can be
> wrong. If you are the creator and disagree, you can appeal this assessment.

> ✍️ Likely human-written — Our automated analysis found no strong signs of AI generation in this content
> (about an **88%** likelihood it was written by a human). This is an automated estimate and can still be
> wrong.

A **Verified-Human creator** (stretch S2) gets a badge prepended: `✅ Verified Human Creator — this creator
completed an identity/authorship verification step. (Detection still runs; the badge is provenance context,
not an exemption.)`

## Appeals workflow

A creator who believes they were misclassified can contest the verdict.

```bash
curl -s -X POST http://127.0.0.1:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-ID", "creator_reasoning": "I wrote this myself. I am a non-native English speaker and my style reads more formal than typical."}'
```

On receipt the system:
1. Looks up the `content_id` (unknown ID → `404`).
2. Flips the content's **`status` to `under_review`** in SQLite and stores the creator's reasoning.
3. Writes an **`appeal` event** to the audit log that references the **original decision snapshot**
   (verdict, confidence, all signal scores).
4. Returns a confirmation with an `appeal_id` and that original snapshot, so the creator sees what they're
   contesting.

There is **no automated re-classification** — an appeal flags the item for a human. Reviewers read the queue
via `GET /appeals`, which returns every `under_review` item with its original scores and the creator's
reasoning.

## Rate limiting

Implemented with **Flask-Limiter** (in-memory store), keyed on client IP.

| Endpoint | Limit | Reasoning |
|----------|-------|-----------|
| `POST /submit` | **10 / minute; 100 / day** | A real creator submits their *own* work occasionally. Even a prolific writer rarely posts more than ~100 pieces a day, and sustained bursts above ~10/min are scripted abuse, not authorship. The per-minute cap stops flooding; the per-day cap stops slow-drip scraping of the classifier. |
| `POST /appeal` | **5 / minute** | Appeals are human-paced and rare; nobody legitimately files more than a handful a minute. A tighter cap discourages spamming the review queue. |

When a limit is exceeded the endpoint returns **HTTP 429** with a JSON body:

```json
{ "error": "rate limit exceeded", "detail": "10 per 1 minute" }
```

**Observed behaviour** (12 rapid `POST /submit` requests against the 10/min limit):

```
request 1  -> 200
...
request 10 -> 200
request 11 -> 429
request 12 -> 429
```

(If earlier submissions already consumed part of the same rolling minute window, the 429s begin
correspondingly sooner — the limiter counts all `/submit` calls per IP in the window.)

## Audit log

Every decision is written to a structured **SQLite** database (`provenance_guard.sqlite`), surfaced as JSON
via `GET /log`. Two tables:

- **`decisions`** — one row per submission: `content_id`, `creator_id`, `content_type`, `timestamp`,
  `attribution`, `confidence`, `p_ai`, the individual signal scores (`p_llm`, `p_style`, `p_lex`,
  `p_meta`), `disagreement`, `status`, and `appeal_reasoning`. Raw text is never stored — only a SHA-256
  hash + a short preview (PII-safe).
- **`events`** — append-only stream of `classified` / `appeal` / `verified` events, so the full history of
  any content is replayable.

**Sample (`GET /log`, ≥3 structured entries; note the appealed item is `under_review`):**

```json
{
  "entries": [
    {
      "content_id": "ed1aa385-…", "creator_id": "erin", "content_type": "text",
      "timestamp": "2026-06-29T04:06:20.365Z",
      "attribution": "uncertain", "confidence": 0.476,
      "p_llm": 0.8, "p_style": 0.61, "p_lex": 1.0, "p_meta": null,
      "status": "under_review",
      "appeal_reasoning": "This is my own writing, I just write formally."
    },
    {
      "content_id": "e57d32a6-…", "creator_id": "img-user", "content_type": "image_metadata",
      "timestamp": "2026-06-29T04:05:29.526Z",
      "attribution": "likely_ai", "confidence": 0.555,
      "p_llm": null, "p_style": 0.6, "p_lex": 0.55, "p_meta": 0.95,

    },
    {
```

## Stretch features

All four stretch features are implemented.

**S1 · Ensemble detection** — the core pipeline already uses **three** weighted signals
(`0.50 / 0.30 / 0.20`) plus a **disagreement-based vote**: when the signals contradict each other the
verdict is forced to `uncertain`, so the ensemble is more than a naive average.

**S2 · Provenance certificate ("Verified Human")** — `POST /verify` issues a credential. The creator
completes a verification step (an original-passage challenge of ≥25 words); the passage is run through
detection and, if it does **not** read as AI, a `certificate_id` is stored against their `creator_id`. Every
future submission by that creator carries a **✅ Verified Human Creator** badge prepended to its label.
Detection still runs — the badge is provenance *context*, not an exemption.

**S3 · Analytics dashboard** — `GET /analytics` (JSON) and `GET /dashboard` (HTML) report total
submissions, verdict distribution, **appeal rate**, average confidence, signal-agreement rate, verified
creators, and — the extra metric of choice — the **AI-verdict appeal rate**, a direct proxy for the
false-positive rate that ties back to this project's guiding principle.

**S4 · Multi-modal support** — `POST /submit` accepts `content_type: "image_metadata"` alongside `text`.
The image path runs a **`metadata_signal`** that reads generative fingerprints (a software/generator tag
like `Midjourney`, an embedded text-to-image prompt, or a C2PA credential) and blends it (dominant weight)
with the text signals run over the image's description. Everything downstream — scoring, labels, appeals,
audit log — is shared. Example: an image tagged `{"software": "Midjourney v6"}` → `likely_ai`, `p_meta 0.95`;
a `Canon EOS R5` photo with no AI markers → `likely_human`.

## Known limitations

The system will **misclassify formal, non-native, or academic human writing** most readily, and it will
under-call it rather than confidently clear it. Concretely: a paragraph of polished academic prose (steady
sentence rhythm, no contractions, connective transitions, sophisticated vocabulary) trips **two of the three
signals** — the stylometric signal reads its low burstiness as "uniform = AI", and the lexical signal reads
its lack of contractions and formal register as AI-like. Only the LLM signal pushes back. The result is that
genuinely human academic writing lands in `uncertain` rather than `likely_human` (see the lower-confidence
example above, where the monetary-policy paragraph scored `p_style 0.81`, `p_lex 0.55`). This is a direct
consequence of *what the stylometric and lexical signals actually measure* — surface uniformity and register,
which formal humans and AI genuinely share. The appeal workflow exists precisely so these creators have
recourse, and the asymmetric thresholds keep the system from making a confident wrong accusation.

A second weak spot: **very short submissions** (under ~40 words) carry too little statistical signal for
stylometrics to be reliable, so the system applies a short-text guard and returns `uncertain` regardless of
the LLM's opinion — it would rather decline to judge than guess on thin evidence.

## Spec reflection

**One way the spec helped.** Writing the [planning.md](planning.md) §2 *uncertainty representation* section
**before any code** forced me to decide what a confidence number should *mean to a user* up front. Because I
had already committed to "confidence = distance from a coin-flip, damped by signal disagreement," the scoring
implementation had a concrete target instead of being reverse-engineered from whatever the signals happened
to output. The disagreement-as-honesty idea — the single most important behaviour in the system — came
directly out of that pre-work, not out of debugging.

**One way the implementation diverged from the plan, and why.** My plan specified the `likely_ai` verdict as
`p_ai ≥ 0.70 AND confidence ≥ 0.50`. In testing this combination turned out to be **practically
unreachable**: a three-signal weighted average rarely exceeds ~0.75, and the confidence formula then
requires near-unanimous 0.9s, so clearly-AI text kept landing in `uncertain`. I replaced the dual gate with
**asymmetric `p_ai` bands (AI ≥ 0.65, human ≤ 0.38) plus a separate disagreement override**. This preserves
the plan's *intent* — it's still structurally harder to accuse a human than to clear one, and signal
disagreement still forces honesty — but it's actually reachable across the verdict range. I left the original
thresholds in planning.md rather than silently rewriting them, so the divergence is visible.

## AI usage

I used an AI coding assistant throughout, directing it against sections of `planning.md` and then verifying
and correcting its output. Two specific instances:

**1. Confidence-scoring function — overrode the verdict thresholds.**
I directed the AI to generate `combine_signals()` from my planning.md §2 (weights, the
`confidence = 2·|p_ai−0.5|·(1−disagreement)` formula, and the threshold table). It produced a reasonable
implementation that faithfully encoded my *written* spec — including the `p_ai ≥ 0.70 AND confidence ≥ 0.50`
gate. When I ran the calibration corpus through it, clearly-AI text scored `p_ai 0.68` and landed
`uncertain`. I diagnosed that **my own spec was the problem, not the code**, and overrode it with asymmetric
`p_ai` bands + a disagreement override (see [Spec reflection](#spec-reflection)). The lesson: the AI
correctly implemented a flawed spec, so the fix was a design decision I had to make, not a bug for the AI to
find.

**2. Groq classifier prompt — revised to kill a false positive.**
I directed the AI to write the LLM signal's prompt to "assess whether text is human or AI and return a JSON
probability." The first version rated the spec's *formal academic human* sample at **0.7** — a false
positive, treating formality as evidence of AI. I overrode the prompt with an explicit **calibration rule**:
"formality, sophistication, and non-native phrasing are NOT by themselves evidence of AI; only lean toward AI
on genuine hallmarks like hedging boilerplate and empty transitions." That single change dropped the academic
text's score from **0.7 → 0.4**, moving it correctly into `uncertain`. I kept the AI's JSON-handling and
error-degradation scaffolding but rewrote the judgement criteria myself.

(Other AI-assisted work — Flask route scaffolding, the SQLite helpers, the HTML dashboard — was reviewed and
edited before use; these two are the instances where I substantively overrode the output.)

## Testing

```bash
python -m tests.test_calibration     # 14 offline deterministic checks
```

The calibration suite verifies signal ordering (AI text scores more AI-like than human text on each
signal), that all three verdict buckets are reachable, that disagreement lowers confidence, that the
short-text guard and degraded-mode confidence cap fire, and that the full pipeline buckets the labelled
corpus correctly. It uses fixed LLM scores so it is deterministic and costs no API quota; the live LLM signal
is exercised by the `curl` examples above.
